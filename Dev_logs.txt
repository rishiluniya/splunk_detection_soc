Week 0
I am currently using a Linux system(Ubuntu 22.04). As I am aiming to work with Windows logs, I set up 2 VMs. A Kali Linux and a Windows 10 system. Initially I wanted the windows system to monitor the logs using splunk but in the industry generally an admin computer manages all the logs collected from different system, so for our demo our host system will act as the log manager, Windows System will be the victim and Kali will be the attacker. 


Week 1 and Week 2
The first thing I did was understand how Splunk looks at logs. I spent a lot of time understanding how indexes work and how the SPL (Search Processing Language) works. For the implementation part I started with installing Splunk on our host Linux. I am working with the trial version of the Splunk Enterprise. Installing was pretty smooth. It works as a server on your system and it runs on a specific port and you can access the GUI using a web browser. I ran it on the default port of 8000. Then I set up a receiving port (9997) so that Splunk receives data. 
Then on the Windows 10 system I installed the Splunk Universal Forwarder and I found out about conf files that need to be established. The outputs.conf decided where to send the data and the inputs.conf decide what logs to monitor. Learning about the conf files was important as they have very specific syntax and I got a default file from the internet and modified it. For collecting logs I was using sysmon and I found out that you need a configure file that tells it what needs to be monitored like: Process creation, Network connections, File creation, etc. The next step was to turn on PowerShell logging using gpedit. There I realized that my Windows 10 Home Edition does not have the provision to access gpedit.msc. I tried to look at alternatives but those were either too complicated or not reliable. So I deleted the entire VM and upgraded to Windows 10 Pro version and redid all the steps. After doing all this I restarted everything and tested it out.
Here I ran into another problem because Splunk wasn't showing any logs in its database. After debugging I realized that I had forgotten to create an index where all the data should be directed(silliest mistake!). After configuration, finally all the data from windows was being sent and I could see the Event codes and I tested some queries which were giving correct outputs. I tried to understand the different Event Codes and how they were categorized. Sysmon categorizes events differently and has number from 1-29 while Windows has codes from 4000-5000 (at least for our project). In our project we have tried to use both the types of codes depending on the situation. Another important caveat of Splunk is making dashboards. Dashboards are a good way to visualize the log data. They can be modified in a plethora of ways and you can use some very neat tables, graphs and charts. I made a very basic dashboard that was not at all optimized but it did help me understand how you can convert very technical logs into graphical data that can be easily understood by everyone. 

Week 3
After learning a lot about Splunk (it never ends!!! ), its time to simulate some attacks. Well first some obvious very simple problems with VMs. I always used VMs with NAT for internet connectivity. There is one small issue here that the IP addresses of these VMs are all identical or to put it in a better way, the IP address(10.0.2.15 in my case)is the default VirtualBox NAT address that every VM gets for itself. So what's the issue here, well if i try to ping from one VM to another, the VM from which I am pinging will try to ping itself. Now I know there may be a way to fix this but since our project is simple and we don't need the machines to be connected to the internet, I just turned it into a Host-Only Network meaning that these machines will only talk between themselves and my host Linux. These proved to be a blessing in disguise later on as the unnecessary noise from the Internet is cut off and the logs are much cleaner. So now my Kali can easily attack the Windows Machine. 
So for our first attack we start with the ever so famous nmap port scan. I know it is very basic but our job here is to not actually penetrate the system but to see how logs look like and how we can identify these attacks. Also attacks like these are called precursors(an observable occurrence or sign that an attacker may be preparing to cause an incident, allowing organizations to identify potential threats before they materialize) and they provide a sort of warning that an attack can happen in the future. The logs caught several attempts to connect to different port (eg 445) in a short span. This was a successful detection and can be used to alert a team that an attack may happen after this.
Next we move to our famous brute force attacks for password cracking. I used Hydra and made a custom wordlist. I chose Remote Desktop Access because it was pretty straight forward and more importantly easy to detect but the issue here was that I was using Windows as the victim and it is pretty secure and forgot to disable it's defenses. In this case it was the Network Level Authentication(NLA) which requires authentication to be completed before the RDP session is established essentially and Hydra's RDP module cannot complete NLA handshakes. There were some other small issues like the configuration file was not configured to log failed authentications and the Events were not appearing in the correct log source but these were an easy fix. After all that the attack was simulated over the current user and another testuser both the user passwords were cracked and I could connect to them over RDP. The logs were generated successfully and we could seperate them from normal local login. 
Next an interesting one Reverse Shell using a payload. Basically downloading a suspicious program and running it. I also wanted to add the a social engineering angle to it and I wanted it to be something attractive and whats more attractive in our current world than RAM after their prices have skyrockted. So I made a webpage that said download free RAM and added a downloadable payload names RAMBooster.exe, Who can give up such an amazing offer! am I right? After the victim downloads the payload and runs it a reverse shell session runs in the attackers terminal and he can now execute commands in that system. For its detection Splunk must first detect a process execution(RAMBooster.exe) and check for a C2(Command and Control) connection which basically means a process that is running commands from outside the local system and also detect suspicious process creation. This was successfully detected by splunk and attacks like these can be quickly dealt with. 

Week 4
After getting hold of the victims computer the attacker should make sure that they have persistent access to the computer even after it reboots and to also escalate privilege. So the defense team should make sure to look at new processes being created and new administrator accounts being created. For example lets say the victim user is infected the attacker can schedule a task that runs our malicious payload (RAMBooster.exe) after every login so the attacker can get access to a terminal on a user with administrative privileges. The defense team looks at splunk logs that show when a task is scheduled and more importantly who created it. For example a non-administrative employee creating a task out of business hours should be flagged as suspicious and should be looked into. 
So, lets say the process is detected but before that you got access to a admin account. Usually killing the process and rebooting will solve the problem for defenders, They might even reset the entire account so that any hidden processes are killed. So our attacker creates a separate admin account and names it something legitimate like 'svcbackup', this provides a fail safe in case the compromised accounts are sanitized. So the defenders need to make sure that all the processes are checked including creation of new accounts and delete them as well. 
This was the last simulated attack and defense for the project
